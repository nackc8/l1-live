[1mDESCRIPTION[0m[m
       Create a new commit containing the current contents of the index and the given log message[m
       describing the changes. The new commit is a direct child of HEAD, usually the tip of the[m
       current branch, and the branch is updated to point to it (unless no branch is associated with[m
       the working tree, in which case HEAD is "detached" as described in [1mgit-checkout[0m(1)).[m
[m
       The content to be committed can be specified in several ways:[m
[m
        1. by using [1mgit-add[0m(1) to incrementally "add" changes to the index before using the [4mcommit[24m[m
[7m Manual page git-commit(1) line 1 (press h for help or q to quit)[27m[K[K           command (Note: even modified files must be "added");[m
[m
        2. by using [1mgit-rm[0m(1) to remove files from the working tree and the index, again before[m
           using the [4mcommit[24m command;[m
[m
        3. by listing files as arguments to the [4mcommit[24m command (without --interactive or --patch[m
           switch), in which case the commit will ignore changes staged in the index, and instead[m
           record the current content of the listed files (which must already be known to Git);[m
[m
        4. by using the -a switch with the [4mcommit[24m command to automatically "add" changes from all[m
           known files (i.e. all files that are already listed in the index) and to automatically[m
           "rm" files in the index that have been removed from the working tree, and then perform[m
           the actual commit;[m
[m
        5. by using the --interactive or --patch switches with the [4mcommit[24m command to decide one by[m
           one which files or hunks should be part of the commit in addition to contents in the[m
           index, before finalizing the operation. See the ‚ÄúInteractive Mode‚Äù section of [1mgit-add[0m(1)[m
           to learn how to operate these modes.[m

[m

[1mEXAMPLES[0m[m
       When recording your own work, the contents of modified files in your working tree are[m
       temporarily stored to a staging area called the "index" with [4mgit[24m [4madd[24m. A file can be reverted[m
       back, only in the index but not in the working tree, to that of the last commit with [1mgit[0m[m
       [1mrestore[0m [1m--staged[0m [1m<file>[0m, which effectively reverts [4mgit[24m [4madd[24m and prevents the changes to this[m
       file from participating in the next commit. After building the state to be committed[m
       incrementally with these commands, [1mgit[0m [1mcommit[0m (without any pathname parameter) is used to[m
[7m Manual page git-commit(1) line 313 (press h for help or q to quit)[27m[K[K       record what has been staged so far. This is the most basic form of the command. An example:[m
[m
           $ edit hello.c[m
           $ git rm goodbye.c[m
           $ git add hello.c[m
           $ git commit[m
[m
       Instead of staging files after each individual change, you can tell [1mgit[0m [1mcommit[0m to notice the[m
       changes to the files whose contents are tracked in your working tree and do corresponding [1mgit[0m[m
       [1madd[0m and [1mgit[0m [1mrm[0m for you. That is, this example does the same as the earlier example if there[m
       is no other change in your working tree:[m
[m
           $ edit hello.c[m
           $ rm goodbye.c[m

[1mDISCUSSION[0m[m
       Though not required, it‚Äôs a good idea to begin the commit message with a single short (less[m
       than 50 character) line summarizing the change, followed by a blank line and then a more[m
       thorough description. The text up to the first blank line in a commit message is treated as[m
       the commit title, and that title is used throughout Git. For example, [1mgit-format-patch[0m(1)[m
       turns a commit into email, and it uses the title on the Subject line and the rest of the[m
[7m Manual page git-commit(1) line 433 (press h for help or q to quit)[27m[K[K       commit in the body.[m
[m
       Git is to some extent character encoding agnostic.[m
[m
       ‚Ä¢   The contents of the blob objects are uninterpreted sequences of bytes. There is no[m
           encoding translation at the core level.[m
[m
       ‚Ä¢   Path names are encoded in UTF-8 normalization form C. This applies to tree objects, the[m
           index file, ref names, as well as path names in command line arguments, environment[m
           variables and config files ([1m.git/config[0m (see [1mgit-config[0m(1)), [1mgitignore[0m(5),[m
           [1mgitattributes[0m(5) and [1mgitmodules[0m(5)).[m
[m
           Note that Git at the core level treats path names simply as sequences of non-NUL bytes,[m
           there are no path name encoding conversions (except on Mac and Windows). Therefore, using[m
           non-ASCII path names will mostly work even on platforms and file systems that use legacy[m
           extended ASCII encodings. However, repositories created on such systems will not work[m
           properly on UTF-8-based systems (e.g. Linux, Mac, Windows) and vice versa. Additionally,[m
           many Git-based tools simply assume path names to be UTF-8 and will fail to display other[m
           encodings correctly.[m
[m
       ‚Ä¢   Commit log messages are typically encoded in UTF-8, but other extended ASCII encodings[m
           are also supported. This includes ISO-8859-x, CP125x and many others, but [4mnot[24m UTF-16/32,[m
           EBCDIC and CJK multi-byte encodings (GBK, Shift-JIS, Big5, EUC-x, CP9xx etc.).[m
[m
[7m Manual page git-commit(1) line 457 (press h for help or q to quit)[27m[K[K       Although we encourage that the commit log messages are encoded in UTF-8, both the core and[m
       Git Porcelain are designed not to force UTF-8 on projects. If all participants of a[m
       particular project find it more convenient to use legacy encodings, Git does not forbid it.[m
       However, there are a few things to keep in mind.[m
[m
        1. [4mgit[24m [4mcommit[24m and [4mgit[24m [4mcommit-tree[24m issues a warning if the commit log message given to it[m
           does not look like a valid UTF-8 string, unless you explicitly say your project uses a[m
           legacy encoding. The way to say this is to have [1mi18n.commitEncoding[0m in [1m.git/config[0m file,[m
           like this:[m
[m
               [i18n][m
                       commitEncoding = ISO-8859-1[m
[m
           Commit objects created with the above setting record the value of [1mi18n.commitEncoding[0m in[m
           its [1mencoding[0m header. This is to help other people who look at them later. Lack of this[m
           header implies that the commit log message is encoded in UTF-8.[m
[m
        2. [4mgit[24m [4mlog[24m, [4mgit[24m [4mshow[24m, [4mgit[24m [4mblame[24m and friends look at the [1mencoding[0m header of a commit object,[m
           and try to re-code the log message into UTF-8 unless otherwise specified. You can specify[m
           the desired output encoding with [1mi18n.logOutputEncoding[0m in [1m.git/config[0m file, like this:[m
[m
               [i18n][m
                       logOutputEncoding = ISO-8859-1[m
[m
[7m Manual page git-commit(1) line 481 (press h for help or q to quit)[27m[K[K           If you do not have this configuration variable, the value of [1mi18n.commitEncoding[0m is used[m
           instead.[m
[m
       Note that we deliberately chose not to re-code the commit log message when a commit is made[m
       to force UTF-8 at the commit object level, because re-coding to UTF-8 is not necessarily a[m
       reversible operation.[m
